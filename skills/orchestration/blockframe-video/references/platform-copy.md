# 平台文案 · 结构模板（YouTube + Bilibili）

`youtube.md` 和 `bilibili.md` 不是自由写作 —— 它们有**固定的段落顺序**，每个平台还不一样。
以前 Step 5 只说「沿用系列模板」却没写模板在哪，于是每一期都靠临场记忆重建，
必然漂移：正文和推广块前后颠倒、章节忘了算片头偏移、B 站套了 YouTube 的结构……
用户每期都得手动改回来。这个文件把顺序**写死**，消灭那轮返工。

> **结构 vs 内容的分界（本文件的红线）**：这里写死的只有**结构、顺序、平台差异、章节偏移规则** ——
> 这些对所有创作者通用。凡是**具体的推广位内容** —— 置顶推广块、赞助行、个人信息里的社群 / 社交 /
> 联系方式、合集或播放列表名 —— 都是**创作者私有**，一律**不写进本 skill**，模板里只留有类型的
> `«占位符»`。填什么由 agent 在生成当期文案时决定，来源按优先级：**当期对话上下文 → agent memory
> 里的创作者档案 → 直接问用户**。**不要在模板或成稿里硬编造任何平台名、品牌、推广方。**

## YouTube · `youtube.md`

段落顺序**从上到下固定**，推广块在正文**之前**（这是 YouTube 的惯例，也是最容易搞反的一处）：

```
## 标题（候选）

1. «钩子» + «系列定位» + «主题名» - «一句白话收益»
2. …（给 2-3 个，第一条对齐系列既有风格）

## 简介

«置顶推广块»            ← 创作者私有，整段照搬，放最前

«个人信息块»            ← 创作者维护的社群 / 社交 / 联系方式清单，逐条照搬

«正文简介»              ← 3-4 段：承接上期一句 → 这期讲什么 → 实打实的点 → 本期独家看点

«系列合集/播放列表名»    ← 纯文字标题，URL 由创作者在后台挂

- «主题名» 源：«主题的原始链接»
- 上一期 «上期主题名»：
«上一期视频链接»        ← 系列内每期都回链上一期；没有就留空让用户补

⏱ 章节
0:00 «开场小标题»
m:ss «各段小标题»       ← 见下方「章节时间戳」

## 标签

«逗号分隔，10-12 个：主题名、作者、平台词、系列词等»
```

## Bilibili · `bilibili.md`

B 站**更精简**：置顶广告位在最前、**不放置顶推广块、不放章节时间戳、不放完整个人信息**。

```
## 标题（候选）
（同 YouTube，可完全一致）

## 简介

«置顶广告位»            ← 创作者私有，B 站描述第一行，整行照搬

«个人信息块（精简）»     ← 只留创作者指定给 B 站的那一两条

«正文简介»              ← 和 YouTube 同一份正文，可略压缩

«主题源链接»

## 标签
«逗号分隔»

## 合集
«系列合集名»            ← 纯文字标题，B 站用合集功能挂
```

### 两平台差异一览

| | YouTube | Bilibili |
|---|---|---|
| 置顶推广块 | 有，放最前 | 无 |
| 置顶广告位 | 无 | 有，第一行 |
| 个人信息块（创作者私有） | 完整列表 | 精简（创作者指定的一两条） |
| 章节时间戳 | 有 | **无** |
| 合集 / 播放列表 | 播放列表名（纯文字） | 合集名 |
| 正文 | 同一份 | 同一份，可略压 |

## 标题公式

系列视频的标题**沿用系列既有口径**，别每期另起风格。观察用户上几期的实际标题，套同一个模：

```
«钩子» «系列定位» «主题名» - «一句白话收益»
```

- **钩子**：数据类最好用（如星数「«作者» «星数»万星」）。数值按发布当天核到的真实值，取整。
- **系列定位**：如「«系列名»之三」，和上期封面 / 标题的编号连上。
- **白话收益**：一句话说清这期能带走什么，避免抽象（「把 spec 拆成 agent 能一张张开工的工单」优于「深入解析 to-tickets」）。
- 备选标题里可以放一条**独家看点**（这期真跑才发现的东西），往往比通用钩子更抓人。

## 章节时间戳 —— 结构由内容规划定，字幕只给起始时间

**章节不是从字幕里「切」出来的。** 章节结构 —— 哪几章、每章讲什么、标题怎么起 —— **由制作时的内容规划决定**：
口播稿的每个 `[NN]` 小节就是一章（见 SKILL.md「章节化生产」的天然单元），标题按该小节主题写。
**字幕只提供一件事：每章的起始时间。** 每章 = 对应 `[NN]` 小节的第一句，起始时间 = 那句在 `narration.srt` 里的 cue 起始秒。

**片头偏移**：若成片加了封面开篇（cover-as-intro），正片被整体后推，每章起始时间要加 offset。

- offset **不是独立常量**，就是 SKILL.md「封面开篇」里的**转场时刻**（对齐开场白结束，约 2.5-3s）—— 以那节为准，本文件不复述数值。
- 正片旁白的 `t` 秒 → 成片的 `t + offset` 秒。
- 首章恒为 `0:00`（封面那几秒算进开场）；其余每章 `= floor(cue_start + offset)`，格式 `m:ss`。
- 不加封面开篇则 offset = 0，直接用 cue 时间。

不加这个偏移，**第二章起**整条章节表会系统性早 2-3 秒 —— 观众点章节全部落在上一句话里。（注意：首章不受影响，仍是 `0:00`，所以「首章是否 0:00」查不出这个错。）

> `narration.srt` 有两个消费者：一个喂动效 `tl`（SKILL.md「SRT 驱动定时」），一个给章节表**查起始时间**。
> 口播稿一改、重跑 TTS，章节结构不变（还是那几个 `[NN]` 小节），但各章起始时间从新 SRT 自动重查，不手工挪。

## 正文简介写法（两平台共用一份）

四段式，承上启下：

1. **承接上期一句**：系列视频假设观众可能没看上期，用一句话把上期结论带出来（「上期我们用 X 把 Y 做了……」）。
2. **这期讲什么**：一句话定位这个主题做的事。
3. **实打实的点**：列 3-4 个这期聊到的具体概念，用短语堆叠，别写成段落。
4. **本期独家看点**：这期真跑 / 真做才碰到的东西（例：真跑一遍才发现 spec 里那条规则会空转）。这一段最值钱，别省。

文风沿用 `personal-chinese-writing-style`：全角标点、单破折号 ` - `、不用网络热词、不用「讲清楚」这类词。
