# 增量重建 · 章节化生产（改一段只重生成一段）

## 动机：一次小改动，实际触发三笔开销

审核口播稿后常有小改动（换个词、调停顿、修一个 TTS 读错的专名）。在**整片单体**流水线里，一个小改动会连锁触发三笔开销：

1. **TTS 全量重生成** —— 哪怕只改一个词，整条旁白重跑一次（云端克隆音 API + credits）。
2. **整片重定时**（最隐蔽）—— TTS **每次重跑，即使句子没变，时间戳也会整体漂移**（TTS 非确定性）。于是所有场景 `data-start` + 所有动效 `tl` 时间全要重新对齐。手工重定时是 agent token 开销的大头。
3. **整片重渲染** —— 4K 十几分钟。

章节化 + 增量重建把三笔都收敛到「只动改了的那一段」。

## 一句话架构

**一个章节 = 一个自包含单元 = { 该段口播 → 该段音频（从 0 开始） → 该段合成 → 渲成一个 clip }。最后按固定参数无损拼接，进度条作为最后一道 overlay 叠上。**

改 `[07]` 只需：重生成 `[07]` 音频 → 该段 SRT 自动重定时 → 只重渲 `[07]` clip → 重新拼接。其余章节的音频与渲染全部命中缓存、不动。

```
narration/07.txt ──TTS──▶ audio/07.mp3 + 07.srt ──▶ build/seg-07/index.html ──render──▶ segs/seg-07.mp4
                                                                                              │
其余段命中缓存(hash 未变) ────────────────────────────────────────────────────────────────┤
                                                                                              ▼
                                        concat(参数一致,无重编码) ──▶ overlay 进度条 ──▶ 片头 ──▶ loudnorm ──▶ 成片
```

## 目录结构

```
<episode>/
├── narration/           # 一段一个文件,喂 TTS 的纯口播
│   ├── 01.txt ... 10.txt
├── audio/
│   ├── 01.mp3 01.srt ... # 每段独立音频 + 字幕(该段从 0 计时)
├── build/
│   ├── seg-01/ ... seg-10/   # 每段一个独立 hyperframes 项目(共享 fonts/kit.css 软链)
├── segs/
│   ├── seg-01.mp4 ...    # 每段渲出的 clip(编码参数严格一致)
├── manifest.json        # 见下:段清单 + hash + 时长 + 缓存状态
└── renders/<slug>-4k.mp4
```

## manifest.json：增量的账本

```json
{
  "fps": 30, "resolution": "landscape-4k", "canvas": [1920,1080],
  "encode": { "vcodec": "libx264", "preset": "slow", "crf": 14, "pix_fmt": "yuv420p", "acodec": "aac", "abitrate": "192k" },
  "chapters": [
    { "id": "01", "narration_hash": "ab12…", "audio": "audio/01.mp3", "srt": "audio/01.srt",
      "seg": "segs/seg-01.mp4", "duration": 43.1, "html_hash": "cd34…" }
  ]
}
```

- **`narration_hash`** = 该段 `narration/NN.txt` 内容哈希。变了 → 重生成音频 + 重渲。
- **`html_hash`** = 该段 `build/seg-NN/index.html` 哈希。仅 HTML 变了(动效/文字改，口播没改) → **只重渲、不重生成音频**（音频、SRT、时长都不变，下游不受影响）。
- 两个 hash 都没变 → 整段命中缓存，`segs/seg-NN.mp4` 直接复用。

重建判定：逐段比对 hash，只对失配的段跑 TTS / 渲染。

## 关键设计点（都可解，但别踩坑）

### 1. 拼接必须无重编码

各段 clip 用**完全一致的编码参数**（`manifest.encode`：分辨率 / fps / pix_fmt / 编码器 / 音频编码）。这样 `ffmpeg -f concat`（demuxer）直接拼、**不重编码**，秒级、零画质损失。参数一旦不一致，concat 要么报错要么强制重编码，增量优势尽失。**渲染每段时把参数从 manifest 读死。**

### 2. SRT 驱动定时（让重定时零手工）

单体流水线里手写死 `tl` 时间是重定时痛点的根源。章节化后每段 SRT 从 0 计时，进一步要求：**动效时间从该段 SRT 自动推导，不写死秒数。**

约定：口播稿里给关键句打锚点标记（如 `⟦q1⟧`），动效绑定「第 N 条 cue」或「某锚点」。构建时读该段 SRT，把 cue 时间填进 `tl`。于是该段 TTS 一变，重跑定时脚本即自动对齐 —— **零手工、零 token**。这条即使不做章节化、单独用在单体流水线，也能立刻消灭重定时开销（见 Gotchas）。

### 3. 持久进度条：作为最后一道 overlay，别让章节各自画

全片进度条是唯一真正跨章节的东西。若让每段各画自己那一截，改一段的时长会让**下游所有段的进度截错位** → 逼着下游重渲。

解法：段渲染时**不画进度条**；拼接完在成片上用一道 ffmpeg `drawbox`（宽度随 `t/总时长` 增长）叠出进度条。与章节彻底解耦，只在总时长变化时重叠一次（一次 encode，几分钟，远快于 HF 重渲）。页眉/页脚是静态的，每段各带一份即可，无继续性问题。

### 4. 章节边界的转场

段内 clip-path 揭幕照旧。跨段边界在 concat 处默认是硬切；让**每段以一个 wipe-in 开场**，边界就是干净的转场而非硬切。要更软可在 concat 处加极短 xfade（会在边界重编码一小段）。

### 5. loudnorm 放最后

响度是整片指标 —— 分段跑会各段基线不一。拼完在**最终整轨**上跑一次 loudnorm（纯音频、便宜）。

### 6. 克隆音跨段一致性

同一克隆音分段合成，偶尔段间能量/韵律有微跳。拼接处加一个 ~60ms 音频 crossfade 抹平即可（`acrossfade`），或接受（多数克隆音读独立段落无碍）。

### 7. 粒度

一段 = 口播稿一个 `[NN]` 小节（天然单元），~10 段。太碎 → TTS/渲染作业与拼接开销上升；太粗 → 增量收益变小。`[NN]` 小节是甜点。

## 收益对照

| 改动 | 单体流水线 | 章节化增量 |
|---|---|---|
| 改一段口播一个词 | 全量 TTS + 全片重定时 + 全片重渲 | 该段 TTS + 该段自动重定时 + 该段重渲 + 秒级拼接 + 一次进度条 overlay |
| 只改一段动效/文字（口播没动） | 全片重渲 | 该段重渲（音频/时长不变，下游全缓存） |
| 只调进度条/片头 | 全片重渲 | 仅最后 overlay/片头那道 encode |

## 与现有流水线的关系

- 上游仍是 `listenhub-tts`，只是**按段调用**（每段一个 `narration/NN.txt`）。
- 段内合成仍是 `producing-video` 那套铁律（挂 `tl`、clip-path 揭幕、字体本地……），只是**规模缩到一段**。
- 全比例封面、平台文案、清单验收不变。
- 拼接 + 进度条 overlay + 片头 + loudnorm 由 `scripts/assemble-episode.sh` 收口。
